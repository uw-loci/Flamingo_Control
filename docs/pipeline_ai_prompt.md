# Flamingo Pipeline JSON — AI Generation Reference

This is a self-contained reference for AI assistants (Claude, ChatGPT, etc.) to generate valid Flamingo microscope pipeline JSON from natural language requests. No codebase access is needed — everything required is in this document.

---

## WARNING — MUST INCLUDE IN OUTPUT

Every AI-generated pipeline MUST include this warning in the response to the user:

> **WARNING: Review this pipeline carefully before running it.** This pipeline was generated by an AI assistant and has not been validated against a live microscope. Pipelines can trigger stage movements and acquisition workflows. Verify all node configurations, connections, and workflow template paths before execution.

---

## JSON Schema

### Top-Level Object

```json
{
  "format_version": "1.0",
  "name": "<pipeline name>",
  "nodes": [ <node>, ... ],
  "connections": [ <connection>, ... ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `format_version` | string | yes | Always `"1.0"` |
| `name` | string | yes | Human-readable pipeline name |
| `nodes` | array | yes | List of node objects |
| `connections` | array | yes | List of connection objects |

### Node Object

```json
{
  "id": "<uuid4>",
  "node_type": "<NODE_TYPE>",
  "name": "<display name>",
  "inputs": [ <port>, ... ],
  "outputs": [ <port>, ... ],
  "config": { ... },
  "x": <float>,
  "y": <float>
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | UUID4 — must be unique across entire pipeline |
| `node_type` | string | yes | One of the valid node type enum values |
| `name` | string | yes | User-visible display name |
| `inputs` | array | yes | Input port objects (exact set depends on node_type) |
| `outputs` | array | yes | Output port objects (exact set depends on node_type) |
| `config` | object | yes | Type-specific config (use `{}` if none) |
| `x` | float | yes | X canvas position (use ~250px spacing between nodes) |
| `y` | float | yes | Y canvas position (typically 100.0 for single-row layouts) |

### Port Object

```json
{
  "id": "<uuid4>",
  "name": "<port name>",
  "port_type": "<PORT_TYPE>",
  "direction": "INPUT" | "OUTPUT",
  "required": true | false
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | UUID4 — must be unique across entire pipeline |
| `name` | string | yes | Exact port name (see per-node tables below) |
| `port_type` | string | yes | One of the valid port type enum values |
| `direction` | string | yes | `"INPUT"` or `"OUTPUT"` |
| `required` | bool | yes | Whether this input must be connected |

### Connection Object

```json
{
  "id": "<uuid4>",
  "source_node_id": "<uuid4 of source node>",
  "source_port_id": "<uuid4 of output port on source node>",
  "target_node_id": "<uuid4 of target node>",
  "target_port_id": "<uuid4 of input port on target node>"
}
```

## Valid Enum Values

### Node Types

```
WORKFLOW
THRESHOLD
FOR_EACH
CONDITIONAL
EXTERNAL_COMMAND
SAMPLE_VIEW_DATA
```

### Port Types

```
VOLUME          — 3D numpy array
OBJECT_LIST     — List of detected objects
OBJECT          — Single detected object (from ForEach)
POSITION        — Stage coordinates (x, y, z, r)
SCALAR          — Numeric value
BOOLEAN         — True/False
STRING          — Text value
FILE_PATH       — Path to a file on disk
TRIGGER         — Execution ordering only, no data payload
ANY             — Accepts/produces any type (pass-through)
```

### Port Direction

```
INPUT
OUTPUT
```

## Complete Config Schemas Per Node Type

### WORKFLOW Config

| Key | Type | Default | Constraints | Description |
|-----|------|---------|-------------|-------------|
| `template_file` | string | `""` | File path (`.txt`) | Path to workflow template file |
| `use_input_position` | bool | `true` | — | If true, use position from input port instead of current stage |
| `auto_z_range` | bool | `false` | — | If true, derive Z-range from input object bounding box |
| `auto_tiling` | bool | `false` | — | If true, auto-compute tiling from object XY extent vs. camera FOV. Single-FOV objects stay 1x1; larger objects get NxM grid with buffer |
| `buffer_percent` | float | `25.0` | >= 0 | Extra buffer around bounding box (%) for Z-range and XY tiling |

### THRESHOLD Config

| Key | Type | Default | Constraints | Description |
|-----|------|---------|-------------|-------------|
| `gauss_sigma` | float | `0.0` | >= 0 | Gaussian blur sigma (0 = no blur) |
| `opening_enabled` | bool | `false` | — | Enable morphological opening |
| `opening_radius` | int | `1` | >= 1 | Opening structuring element radius |
| `min_object_size` | int | `0` | >= 0 | Minimum object size in voxels |
| `default_threshold` | int | `100` | 0-65535 | Fallback threshold |
| `channel_thresholds` | object | `{}` | Keys are **strings**: `"0"`, `"1"`, `"2"`, `"3"` | Per-channel threshold values |
| `enabled_channels` | array | `[0,1,2,3]` | List of ints 0-3 | Which channels to analyze |

**IMPORTANT:** `channel_thresholds` keys must be JSON **strings** (`"0"`, `"1"`), not integers. This is a JSON serialization requirement.

### FOR_EACH Config

Empty object: `{}`

### CONDITIONAL Config

| Key | Type | Default | Valid Values | Description |
|-----|------|---------|--------------|-------------|
| `comparison_op` | string | `">"` | `">"`, `"<"`, `"=="`, `"!="`, `">="`, `"<="` | Comparison operator |
| `threshold_value` | float | `0.0` | any number | Value to compare against |

### EXTERNAL_COMMAND Config

| Key | Type | Default | Valid Values | Description |
|-----|------|---------|--------------|-------------|
| `command_template` | string | `""` | shell command | Command to execute |
| `input_format` | string | `"numpy"` | `"numpy"`, `"tiff"`, `"json"` | How input data is serialized to disk |
| `output_format` | string | `"json"` | `"json"`, `"csv"`, `"numpy"` | How output data is read from disk |
| `timeout_seconds` | int | `300` | > 0 | Max execution time before kill |

### SAMPLE_VIEW_DATA Config

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `channel_0` | bool | `true` | Include 405nm (DAPI) channel |
| `channel_1` | bool | `true` | Include 488nm (GFP) channel |
| `channel_2` | bool | `true` | Include 561nm (RFP) channel |
| `channel_3` | bool | `true` | Include 640nm (Far-Red) channel |

## Port Type Compatibility Matrix

A connection from source→target is valid only if the pair appears below.

| Source Type | Valid Target Types |
|-------------|-------------------|
| `VOLUME` | `VOLUME`, `ANY` |
| `OBJECT_LIST` | `OBJECT_LIST`, `ANY` |
| `OBJECT` | `OBJECT`, `POSITION`, `ANY` |
| `POSITION` | `POSITION`, `ANY` |
| `SCALAR` | `SCALAR`, `BOOLEAN`, `ANY` |
| `BOOLEAN` | `BOOLEAN`, `ANY` |
| `STRING` | `STRING`, `FILE_PATH`, `ANY` |
| `FILE_PATH` | `FILE_PATH`, `STRING`, `ANY` |
| `TRIGGER` | **any type** (TRIGGER provides execution ordering) |
| `ANY` | **any type** |

---

## Connection Rules

1. Each input port can have **at most one** incoming connection
2. Output ports can have **multiple** outgoing connections
3. No self-connections (source and target must be different nodes)
4. The graph must be a **DAG** (no cycles)
5. Source must be an OUTPUT port, target must be an INPUT port
6. Port types must be compatible per the matrix above

---

## UUID Requirements

- All `id` fields must be valid UUID4 format: `xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx` where `y` is `8`, `9`, `a`, or `b`
- Every ID in the pipeline must be globally unique (node IDs, port IDs, connection IDs — all share one namespace)
- Generate fresh UUIDs for every pipeline — never reuse IDs across different pipelines

---

## Layout Conventions

- Space nodes ~250px apart horizontally (x axis)
- Use y=100.0 for single-row layouts
- For branching (e.g., Conditional), offset branches vertically (y=50 and y=200)
- Source/data nodes on the left, processing in the middle, sinks on the right

## Common Patterns

### Data Source → Analysis
Start with `SAMPLE_VIEW_DATA` (existing data) or `WORKFLOW` (new acquisition), then connect volume output to `THRESHOLD` input.

### Detect and Iterate
`THRESHOLD.objects` → `FOR_EACH.collection` → `FOR_EACH.current_item` → `WORKFLOW.position` (with `use_input_position: true`).

### Conditional Branching
Connect any `SCALAR` output (like `THRESHOLD.count`) to `CONDITIONAL.value`. The `true_branch`/`false_branch` TRIGGER outputs control which downstream nodes execute.

### Chained Acquisitions
Connect `WORKFLOW.completed` (TRIGGER) to the next `WORKFLOW.trigger` to sequence acquisitions.

## Pre-Output Checklist

Before outputting a pipeline JSON, verify:

- [ ] All IDs are valid UUID4 format and globally unique within the pipeline
- [ ] Every node has exactly the ports listed in the port definitions for its type
- [ ] Port names match exactly (case-sensitive): `volume`, `objects`, `count`, etc.
- [ ] Port types and required flags match the definitions
- [ ] All connections use existing node IDs and port IDs
- [ ] All connections go from OUTPUT ports to INPUT ports
- [ ] Port type compatibility is satisfied for every connection
- [ ] No input port has more than one incoming connection
- [ ] The graph has no cycles
- [ ] `channel_thresholds` keys are strings (`"0"`) not integers
- [ ] `format_version` is `"1.0"`
- [ ] The safety warning is included in your response to the user

## Saving Pipelines

Pipeline JSON files are saved to `~/.flamingo/pipelines/` with a `.json` extension. The filename is derived from the pipeline name (spaces replaced with underscores, lowercased). To load a generated pipeline:

1. Save the JSON to `~/.flamingo/pipelines/<name>.json`
2. Open the Pipeline Editor (Extensions → Pipeline Editor)
3. Use File → Open to load the pipeline
4. Review all node configurations
5. Run only after verification
